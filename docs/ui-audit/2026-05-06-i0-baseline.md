# UI Audit — Phase I.0 Baseline — 2026-05-07

## Summary

- App version: 0.1.0 (commit cc509e2 + phase-h patches)
- Layout system: **structurally sound.** Root layout follows the
  correct `h-screen → flex → overflow-hidden → overflow-auto` pattern.
- 7 views audited. 2 have structural issues. 5 are correct.

## Root layout (App.tsx)

```
div.flex.flex-col.h-screen.bg-background
  div.flex.flex-1.overflow-hidden
    div.flex.flex-col.w-14 (sidebar)
    div.flex.flex-col.flex-1.overflow-hidden
      Toolbar
      main.flex-1.overflow-auto ← views render here
  StatusBar
```

**Verdict: CORRECT.** `h-screen` constrains height, `overflow-hidden`
prevents double-scrollbar, `main.overflow-auto` is the scroll container.

## Per-view audit

### ChatView — OK
- Root: `div.flex.flex-col.h-full`
- Uses ResizableLayout with `overflow-hidden` panels
- Message list: ScrollArea with flex-1
- No issues.

### InboxView — OK
- Root: `div.flex.flex-col.h-full`
- Message list: `ScrollArea.flex-1`
- No issues.

### TasksView — OK
- Root: `div.flex.flex-col.h-full`
- Board: `div.flex-1.overflow-auto.p-3` with horizontal scroll
- Detail drawer: `ScrollArea.flex-1` inside Sheet
- No issues.

### FilesView — OK
- Root: `div.flex.flex-col.h-full` (empty state) / has `ScrollArea.flex-1`
- No issues.

### NotesView — OK
- Root: `div.flex.flex-col.h-full` (empty state)
- Uses ResizableLayout
- No issues.

### CalendarView — OK (empty state only)
- Empty state: `div.flex.items-center.justify-center.h-full`
- No scroll needed for empty state.

### SettingsView — FLAG (minor)
- Root: `div.flex.flex-col.gap-6.p-6.max-w-2xl`
- **Missing `h-full`.** Content extends naturally downward.
- Scroll handled by parent `main.overflow-auto` — works correctly.
- **Not broken**, but doesn't follow the `h-full` pattern of other views.
  Acceptable because Settings is inherently a scrollable page (not a
  constrained layout like Kanban or Chat).

## Anti-patterns found

1. **SettingsView inconsistency (MINOR):** Only view without `h-full`
   on root. Not broken because parent `main` handles scroll. Cosmetic
   consistency fix only.

2. **No `min-h-0` needed:** Checked all flex children with overflow —
   none are missing `min-h-0` because the views either use `h-full`
   (which sets an explicit height, bypassing the need for `min-h-0`)
   or use `ScrollArea` from radix which handles this internally.
